Project: Books and Brews
Goal: Build Phase 7 Step 2 — deposit + remaining balance system

Context:
This project already has:
- admin CRM
- secure client portal
- invoices
- Stripe payment links
- quote/package system
- project tracker
- project completion system
- file uploads via R2
- client messaging

The package system already stores `deposit_percent` in `service_packages`.
Right now, package creation only creates one invoice. We need to upgrade this into a true deposit + remaining balance workflow.

Your job:
Implement the full deposit system cleanly across backend + admin UI + client portal. If any existing file needs multiple changes, regenerate the entire file instead of patching fragments.

What the deposit system must do:

1. Package-to-invoice logic
When admin creates an invoice from a package:
- create a DEPOSIT invoice first if `deposit_percent > 0`
- calculate deposit amount from package price
- calculate remaining balance
- store enough metadata so remaining balance can be generated later
- if deposit_percent is 0, create a normal full invoice

2. Invoice data model
Support these invoice concepts:
- invoice type: `deposit`, `balance`, or `full`
- total project amount
- deposit percent
- remaining balance amount
- balance invoice relationship to deposit invoice if useful
- payment status should remain compatible with current system

3. Admin workflow
In `admin/manage.html`:
- show invoice type in invoice cards
- show total project amount and remaining balance where relevant
- add a button on paid deposit invoices:
  `Create Remaining Balance Invoice`
- prevent duplicate balance invoice creation for the same deposit/project
- keep existing Stripe generation and resend invoice flow working

4. Remaining balance creation
Create a backend route that:
- takes a deposit invoice or lead/project context
- creates a new balance invoice for the remaining amount
- marks it as type `balance`
- sets notes clearly, like “Remaining balance for [package/project]”
- rejects creation if:
  - deposit invoice is unpaid
  - remaining balance is zero
  - balance invoice already exists

5. Client portal
In `client/portal.html`:
- show invoice type labels like Deposit / Balance / Full
- show clear billing breakdown when available:
  - total project amount
  - deposit paid / due
  - remaining balance
- keep current Pay Now button behavior

6. Stripe compatibility
Do not break existing Stripe invoice generation flow.
Existing “Generate Stripe Payment Link” should continue working with:
- deposit invoices
- balance invoices
- full invoices

7. Database changes
Create or document required SQL changes.
Use D1-compatible SQL.
If new columns are needed, provide ALTER TABLE statements.
Be careful not to destroy existing data.

Suggested invoice fields to support:
- invoice_type
